# GitHub 與 Side Project 統一作業標準

> 目的：建立一套「簡單、乾淨、可長期維護」且不造成焦慮的個人 GitHub 經營與專案管理準則。
> 適用對象：單人 Side Project / 履歷作品集導向 / 長期技術累積
> 核心精神：清晰、一致、可解釋，而非形式完美。

---

## 一、核心心法（最高原則）

1. 清晰 > 完整 > 複雜
2. Commit 是語意紀錄，不是打卡
3. README 是入口，不是百科
4. Study Notes ≠ 成果展示
5. 一致性存在於「思考」，不是「形式」

> 六個月後的自己，能在 1 分鐘內理解專案存在理由，才是高手標準。

---

## 二、最容易後悔的 6 件事（提前避免）

### 1. 把「學習內容」過早當成成果
- 問題：半年後不認同內容，想刪除或隱藏
- 原則：學習 ≠ 成果
- 對策：Study Notes 與 Public 成果嚴格分離

### 2. Repo 名稱綁死技術
- 問題：`react-xxx`、`angular-xxx`，技術更換即過時
- 原則：描述「系統是什麼」，不是「用什麼寫」

### 3. 為了完整而拖延輸出
- 原則：能解釋一個概念，就值得一篇文章

### 4. README 承擔過多責任
- 原則：README 是入口導覽，不是百科全書

### 5. 為了綠點而 commit
- 原則：Commit 是語意紀錄，不是每日打卡

### 6. 過早優化資料夾結構
- 原則：結構為使用服務，不為美感存在

---

## 三、Study Notes 的最終定位

唯一目的：幫助「真正理解」，不是展示理解。

建議策略：
- 預設 Private
- 若 Public：
    - 明確標示為學習筆記
    - 不放履歷、不 pin
    - README 第一屏說明用途

---

## 四、Public 成果的進入門檻（4 項檢查）

滿足以下即可公開：
1. 能用自己的話解釋
2. 知道解決什麼問題
3. 知道不適用情境
4. 有實際或模擬案例

> 不需要完美，只需要負責。

---

## 五、Repo 命名鐵律（避免未來重命名）

規則：
- 全小寫
- 2–4 個單字
- 不含技術名稱
- 可被自然念出

範例：
- study-notes
- qr-ordering-system
- engineering-blog

---

## 六、年度 Roadmap 原則（避免過度規劃）

高手結論：只需要方向，不需要細節。

建議：
- 年度：主題方向
- 季度：產出類型
- 月：彈性調整

---

## 七、專案層級分類（先判斷再套規範）

### 正式專案（必須完整 SOP）
符合任一：
- 履歷或作品集用途
- 維護超過 2 週
- 有架構或技術決策
- 可能擴充或重構

### 實驗 / 練習專案
- 純學習
- 一次性實驗
- 可簡化結構

> 只有正式專案才強制要求 docs/。

---

## 八、正式專案標準結構

```
my-project/
├── .github/
├── docs/
│ ├── STRUCTURE.md
│ ├── DEPENDENCIES.md
│ └── DEVELOPMENT.md
├── src/
├── .env.example
├── .gitignore
├── CHANGELOG.md
├── LICENSE
├── package.json
└── README.md
```

重點：docs/ 記錄「為什麼這樣設計」，不是只列出工具。

---

## 九、README 統一職責（不可超越）

README 只回答五件事：
1. 這是什麼？
2. 為什麼存在？
3. 如何快速啟動？
4. 詳細文件在哪？
5. 專案成熟度？

禁止放入：
- 詳細架構說明
- 長篇指令解析
- 套件完整清單

## 十、docs/ 文件撰寫準則

### STRUCTURE.md
- 資料夾責任
- 命名規則
- 為何這樣分層

### DEPENDENCIES.md
| 套件 | 用途 | 為什麼選擇 | 替代方案 |
|-|-|-|-|

### DEVELOPMENT.md
- 環境需求
- 啟動流程
- 開發規範

## 十一、Commit Message 標準（個人最小集合）

基本格式：
<type>: <verb> <object>

核心 type（最小必要）：
- feature
- fix
- docs

必要時可擴充：
- refactor / chore / test / perf / style

原則：
- 一個 commit = 一個意圖
- 說明「為什麼改」，不用描述每行變動
- 不使用無語意的 `Update xxx`

10 秒規則：
如果猶豫超過 10 秒，直接用：
- docs: update documentation
- feature: add initial implementation
- fix: minor issue

---

## 十二、Branch 與 Git 工作流（單人版）

核心心法：
1. branch 是草稿，不是展示品
2. master/main 只放可展示成果
3. 歷史乾淨比歷史完整更重要

### 必開 branch 情境
- 新功能
- Bug 修復
- 大量文件變更

### 可直接 commit
- 小 typo
- 排版調整

### 推薦流程
開 branch → 完成 → rebase → fast-forward merge → 刪除 branch

一條直線歷史 = 理想狀態。

---

## 十三、知識管理與內容 Repo 分工

- 系統 Repo：部落格平台 / 專案本體
- 內容 Repo：技術文章 / 知識筆記

結構範例：
```
articles/
├── javascript/
├── angular/
├── react/
└── system-design/
```

Issue 用途：靈感、草稿、TODO（非正式內容）。

---

## 十四、例外與推翻條款（防止僵化）

允許破例需同時滿足：
1. 能說出違反哪條原則
2. 能說明為何這次更好
3. 不影響對外信任敘事
4. 願意留下紀錄

推翻規則的成熟條件：
- 已長期穩定輸出
- 清楚該規則解決過去問題
- 新規則抽象層級更高

---

## 十五、不必過度焦慮的事項

以下不需過度完美：
- 早期 Repo 品質普通
- Commit 紀錄不完美
- 舊文章觀點保守

只要主敘事清楚，其餘都是歷史背景。

---

## 十六、最終自我檢查清單

- 我能一句話說明 Repo 存在理由嗎？
- README 是否只負責入口？
- docs 是否記錄決策而非列表？
- 六個月後我能快速理解嗎？
- Commit 是否具有語意而非打卡？
